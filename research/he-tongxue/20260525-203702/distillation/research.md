## Design DNA
### Core
1. Abstract topics become a tangible invention, prototype, device, experiment, product, or visible data screen.
2. The cover is usually built around one proof object. The object must make the title believable before the viewer reads the video title.
3. The subject is a maker, demonstrator, witness, or scale reference; the face supports the object instead of replacing it.
4. Text is often absent, or physically bound to the object: on a screen, product name, label, price, function word, or direct comparison tag.
5. The frame uses real-camera product photography, macro proof, wide-angle machine depth, or overhead lab composition. It should feel like the thing was actually built.
6. The emotional tone is clean curiosity and maker confidence, not screaming shock, luxury flex, or pure lecture authority.

### Conditional
1. Use strong red/blue polarity when the topic is a real comparison, test, or binary decision.
2. Use huge typography only when it functions as a product name, label, price, or object-bound proof.
3. Object-only covers work when the object itself contains a strong mystery or proof detail.
4. Motion blur and action work when the invention is absurd or kinetic.

### Weak
1. Black studio background appears often, but it is not the engine. It is a proof-isolation tool.
2. Creator face appears often, but the face alone is not enough; ordinary portrait plus title drifts away.
3. "Technology" is a common lane, but the deeper pattern is utility made physical.

## Cover Generation Engine
Engine type: hybrid `Utility Clarity Engine` + `Narrative Suspense Engine` + `Aesthetic Identity Engine`.

Viewer decision compressed: "Is there a real, clever, visually interesting thing here that solves or tests a problem I recognize?"

Topic-to-cover mechanism: first translate the topic into a buildable or observable object, then choose the most convincing proof moment: finished prototype, internal structure, comparison test, strange action, visible data, or product close-up.

Subject role: maker-demonstrator-witness. The subject points, holds, rides, reveals, or stands beside the object to prove scale and ownership.

Pre-visual decision: decide what physical proof the viewer can see in one second. If there is no proof object, no experiment state, and no specific "we made this" moment, the cover will become generic tech poster art.

Drift risk: MrBeast-style extreme stakes, Dan Koe-style text authority, 影视飓风-style cinematic mood, product ad polish without maker proof, or generic AI dashboard collage.

## Topic Translation Rules
- Abstract topics: turn them into a physical machine, prototype, table, screen, experiment rig, workflow desk, strange object, or visible before/after data. Example: "AI content flywheel" becomes a real desk machine that takes messy project folders and outputs finished thumbnails/articles.
- Concrete topics: make the actual object bigger, closer, stranger, or more mechanically legible. Show internals, scale, comparison, or a hand interacting with it.
- Required translation variables: viewer problem, invented solution, proof object, object's physical form, subject action, one-second mystery, optional object-bound label.
- Do not translate abstract topics into floating icons, holograms, generic robots, app dashboards, PPT diagrams, or slogans pasted beside a face.
- If the topic has no obvious physical form, invent a plausible studio prototype that embodies the function.

## Cover Storyboard Rules
- Story moment: the viewer catches the creator at the moment of revealing or using the built thing.
- Visible conflict: everyday problem vs over-engineered solution; real vs fake; slow vs fast; hidden internals vs polished product; boring behavior vs playful machine.
- Subject action: point to the machine, hold two comparison objects, operate a button, ride/use the prototype, open the internals, or look through/behind the device.
- Proof object: the invention itself, a close-up part, an experiment array, a phone screen, a hand-drawn plan, a price tag, or a labeled physical comparison.
- Viewer question: "What is that thing, and why did he build it?"
- Forbidden static compositions: portrait with title, floating UI collage, beautiful but non-functional tech scene, clean product ad with no maker relationship, diagram-only poster, or broad metaphor with no object proof.

## Design Layout Brief Rules
- First read: the proof object or object-bound label.
- Second read: subject action/face or comparison state.
- Third read: small clue such as price, object name, screen data, process sketch, or color polarity.
- Layout zones: reserve 55-75% visual weight for the object. Place the subject at an edge, behind, above, or between objects as proof of scale/ownership.
- Reading path: object first, action/face second, label/detail third.
- Negative space: use black void, clean table surface, white machine surfaces, or simple studio wall to isolate the object.
- Forbidden layouts: central portrait with title blocks; text-first thesis; more than three unrelated props; unrelated badges; object so small it becomes decoration.

## Copy Hierarchy Rules
- Main title: usually not needed. If used, it should be a short product name, object function, price, number, or binary label.
- Subtitle: avoid. Let the video title outside the cover carry nuance.
- State labels: use only for actual object states or comparison sides, such as `快充/慢充`, `真的/假的`, `自动开灯`, `¥49999`.
- Object/zone binding: labels must sit on the object, above the object, inside the screen, or visually attached to the corresponding side.
- Isolation rules: each label belongs to one physical object or side. Use distance, color, or a plate to prevent accidental combined phrases.
- Forbidden adjacency: do not place a floating main title near side labels where the viewer reads them as one sentence.
- Removal rule: if text competes with the object, remove the text and strengthen the object proof.

## Composition Rules
- Prefer macro close-up, wide-angle foreground object, overhead construction view, or two-object comparison.
- Use hands to prove interaction and scale.
- Keep the object physically plausible; screws, cables, screens, keys, rails, workbench surfaces, and realistic reflections matter.
- Use face scale as secondary proof, not as the whole hook.

## Subject Rules
- Use the user's portrait as identity reference, but redesign pose around the prototype.
- Good actions: presenting, pointing, holding, operating, looking through, opening internals, riding/using, or calmly staring while holding comparison objects.
- Bad actions: default thinking pose, arms crossed authority pose, influencer grin with no object relationship, exaggerated shock without proof.

## Identity And Final Prompt Firewall
- Internal research and skill routing may name 何同学, but the final GPT Image 2 generation prompt must not contain `何同学`, `He Tongxue`, `何同学-inspired`, `He Tongxue-inspired`, `in the style of 何同学`, `like 何同学`, or equivalent wording.
- Translate the pattern into concrete rules: real physical proof object, maker-demonstrator role, object-bound short label, studio/product photography, visible mechanism, and clean lab environment.
- If the user supplies a portrait reference, the generated subject must preserve the user's supplied identity traits and must not resemble any public creator. Do not name 何同学 in the final generation prompt.

## Text Rules
- Use large clean sans-serif Chinese or Latin text only when it behaves like product typography or object labeling.
- Screen text can be oversized and high contrast.
- Keep on-cover copy short: 1-4 Chinese characters per label, or one short product name.
- Prefer labels that name the object state, not full sentence claims.

## Typography Layout System
- Protected text zones are object-bound zones: phone screens, product surface, label area above an object, or clean negative space adjacent to a machine.
- Use thick white text for product names on dark machinery; use red/blue giant type for binary tests; use pink/white outlined labels for playful real/fake app comparisons.
- Do not put a long title in a corner unless the object cannot carry meaning.
- If the background is busy, create a clean plate or move text onto the object screen.

## Color And Lighting Rules
- Dominant palette: clean black, white, graphite, lab gray, product metal, phone-screen glow.
- Accents: cyan/blue, red, yellow, pink, or one product-specific color.
- Lighting: real studio/product lighting, crisp highlights, high local contrast, shallow depth or wide-angle perspective.
- Avoid generic neon AI purple-blue, stock cyberpunk, or over-saturated cartoon lighting.

## Hook Mechanics
- "I/we made this": a real thing exists.
- "For this everyday problem, we built an absurdly specific solution."
- "This product or experiment proves a hidden result."
- "Real vs fake" or "fast vs slow" comparison.
- "Internal structure revealed": the viewer sees inside something usually hidden.
- "Price/quantity/time proof": number attached to a real object or experiment.

## GPT Image 2 Prompt Rules
- Start by naming the physical proof object and subject action before describing style.
- Specify the target canvas and platform.
- Require a real-camera product photography look, not illustration.
- Describe the object with physical parts, scale, surfaces, and interaction.
- State where text appears and what object it is bound to.
- Include negative constraints against generic AI icons, holographic dashboards, text-first poster layouts, and copied 何同学 likeness.

## Avoid
- Copying 何同学's face, channel identity, logos, or exact cover compositions.
- Treating a user's own portrait as a fixed pose reference.
- Abstract "AI" visual shorthand without a real object.
- Long floating titles.
- A beautiful tech background with no proof event.
- MrBeast stakes, Dan Koe authority claims, or 影视飓风 documentary stills.
