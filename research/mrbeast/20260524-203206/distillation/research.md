## Design DNA

### Core

- One visible event beats one abstract concept. The cover should show a thing happening, not explain a system.
- Stakes are physical: money, time, danger, boundary, crowd, luxury object, impossible place, or visible suffering.
- The subject is inside the event. The host is a participant/presenter/witness, not a detached model.
- The premise is child-legible: viewers should know the rule, reward, or danger in under one second.
- Text is sparse and scale-oriented: numbers, `DAY`, price labels, or no text.
- Title, thumbnail, and intro form one promise. The thumbnail must be truthful enough that the video can immediately pay it off.

### Conditional

- Use split-screen for price ladders and before/after extremes.
- Use a red ring, line, platform, cage, coffin, button, or marked zone when the rule is the hook.
- Use crowds when social scale is the hook.
- Use tropical/luxury environments when the click promise is prize fantasy.
- Use storms, dirt, height, confinement, or strained body language when the promise is endurance.

### Weak

- Open-mouth expressions are not universally required; process evidence notes closed-mouth tests can perform better in some cases.
- Blue/red/gold palettes recur but are secondary to premise clarity.
- Putting a face in every thumbnail is partly brand/trust logic and should not be copied literally when using a user's own identity.

## Cover Generation Engine

MrBeast uses a `Stakes Engine`.

The cover compresses this viewer decision:

> Is this extreme, valuable, dangerous, absurd, or clear enough that I need to see the outcome?

The engine works in this order:

1. Start from an idea with broad curiosity or desire.
2. Translate it into one physical stake: prize, cost, time, danger, boundary, crowd, luxury object, or impossible scale.
3. Make the rule/result visible as a photographed event or object.
4. Put the host or subject inside that event.
5. Reduce the cover to one dominant event and one optional scale label.
6. Ensure the first seconds of the video can pay off the promise.

This is not a generic "loud thumbnail" engine. It is a promise-and-proof engine.

## Topic Translation Rules

- Abstract topics must be embodied. Do not draw a conceptual diagram unless it is made into a physical machine, prize, rule, or challenge object.
- Systems become machines or arenas. A "content flywheel" becomes a giant content machine or game loop that visibly produces videos, views, money, or followers.
- Productivity becomes time pressure, output quantity, or failure vs success.
- Business/wealth topics become price ladders, money piles, luxury objects, or extreme before/after states.
- Learning/knowledge topics become challenges, experiments, or visible proof scenes.
- Technology topics become physical tech objects, vehicles, laboratories, robots, spaceships, or impossible demonstrations.
- Avoid translations that drift into Dan Koe/authority logic: calm portrait, minimal concept card, elegant quote, abstract schema.
- Avoid translations that drift into utility tutorial logic: UI dashboard, list of steps, SaaS feature board.
- Forbidden for MrBeast-derived prompts: pure flowchart, clean system diagram, lecture slide, knowledge-course poster, calm expert portrait, generic futuristic interface.

## Composition Rules

- One focal event first, decoration second.
- Use large foreground faces or bodies only when they strengthen the event.
- Use split-screen only when the two sides are extreme enough to read instantly.
- Use visible boundaries: red circles, hazard platforms, cages, rooms, coffins, islands, vehicles.
- Use crowd geometry to show scale, but keep crowd identities simple enough for mobile.
- Avoid micro-detail. If an element cannot be read at phone size, it should not carry the premise.

## Subject Rules

- The host should not pose as a fashion model or calm expert.
- The host should be participating, presenting, reacting, pushing, holding, surviving, pointing, or witnessing.
- Facial expression must match the event: delight for prize fantasy, strain for survival, shock for danger, confidence for comparison.
- When using the user's own portrait, preserve identity traits only; redesign action and expression for the event.

## Text Rules

- Use no text when the event itself is obvious.
- When text is used, prefer one numeric scale label, progress marker, or two matched comparison labels.
- Text should not explain the whole title.
- Examples: `$1`, `$1M`, `$500,000`, `DAY 97`, `0 -> 100`, `1 vs 100`.
- Avoid abstract labels as the main hook unless the visual event has already done the work.

## Typography Layout System

- Treat text as a protected layer.
- Put text on clean negative space, a badge, or a simple backing plate.
- For split comparisons, matched labels must share size, stroke, padding, and placement logic.
- Numeric labels should be oversized and thick, usually white with black stroke or yellow/white with dark stroke.
- Progress labels like `DAY 97` can sit in a top safe area over simplified background.
- Do not place text over faces, hands, prizes, dense crowds, complex UI, or the center of the physical stake.
- For Chinese adaptation, use short phrases or numeric labels, not long explanatory copy.

## Color And Lighting Rules

- Use high-contrast, feed-readable commercial realism.
- Saturated red, blue, yellow/gold, green grass, white faces/clothes, and hazard colors recur.
- Luxury side is clean, gold, blue, white, tropical, or glossy.
- Danger/survival side is dirtier, darker, stormy, red, gray, muddy, or cramped.
- Faces and proof objects must be brighter than background clutter.

## Hook Mechanics

- Price ladder: worthless vs absurdly expensive.
- Endurance: survive a visible physical condition for a number of days/hours.
- Boundary rule: stay inside, do not touch, last to leave, fit within, cross or do not cross.
- Prize fantasy: a desirable object/place becomes the reason to click.
- Social scale: many people compete under one simple rule.
- Physical danger: height, confinement, storm, wild environment, isolation.

## GPT Image 2 Prompt Rules

- Begin with `Cover generation engine: Stakes Engine`.
- Include a `Topic translation` section before composition.
- For abstract user topics, explicitly forbid pure diagrams and force physical embodiment.
- Define the single visible stake before describing style.
- Use the user's reference image for identity only; choose a new action from the event.
- State exact on-image text once, and keep it short.
- Include mobile readability and anti-infographic constraints.

## Avoid

- Pure diagrams, system maps, clean SaaS dashboards, or generic AI-tech posters.
- Calm intellectual portraits.
- Decorative MrBeast mimicry without stakes.
- Overcrowded collage where the rule is unclear.
- Long explanatory text.
- Copying MrBeast's face, logo, exact layouts, protected IP, or existing thumbnails.

## User Input Questions

1. Which platform and ratio?
2. What is the raw video topic?
3. What is the desired result or transformation?
4. What is the visible stake: money, time, quantity, danger, reward, failure, crowd, or scale?
5. Should the user's own face appear?
6. What exact text, if any?
7. What must appear and what must not appear?
